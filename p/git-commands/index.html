git的使用，包含日常项目使用的git指令和规范

2025-09-02
阅读时长: 8 分钟

在参加OS内核比赛的时候，我们队最终项目是~1000 commits和10+ branches，可见使用git管理之麻烦。

但也由此机会积累了不少使用git的经验，故此处记录下来，以便日后参考。

虽然完全可以通过git --help或是git <cmd> --help来看如何使用。

初始化仓库

首先，使用git init可以创建一个git仓库，这个指令会在当前目录下面创建一个.git目录，并在里面存放git仓库管理的各类元文件，可以说有了它，当前目录才标识为git管理。如果你移除.git目录，需要重新执行git init指令才能重新将当前目录归于git管理，不过你之前的.git所含有的历史信息会全部丢失。

branches: 早期 Git 的残留目录，现在基本不用了。以前用于实现“非参考方式”下的分支管理。现今版本几乎始终为空，可以忽略。

hooks:包含客户端或服务端的钩子脚本（shell、python等），这些脚本在特定事件（如提交、合并、推送前后）时自动执行。

info:存放一些辅助信息，比如 .git/info/exclude 文件，可以设置本地忽略文件（类似 .gitignore，但只对本地有效，不会提交）。

logs:用于保存对分支和引用（ref）的操作日志。可用于 git reflog 命令查看 HEAD 和各分支的历史指向。例如logs/refs/heads/记录本地分支操作历史，logs/refs/remotes/origin/记录远程分支操作历史。

objects:存储所有 Git 对象（数据的“内容”和历史），包括 commit（提交）、tree（目录）、blob（文件内容）、tag（标签）等对象。每个对象都是用 SHA-1 哈希命名的子目录。这是 Git 核心存储：所有历史内容都在这里！

refs:保存所有引用（ref，即对提交对象的指针）的信息，这些实际控制了你看到的分支和标签。

  • refs/heads/：本地分支，每个分支一个文件，内容是 commit 的 SHA-1 值。
  • refs/remotes/origin/：远程分支跟踪仓库的分支。
  • refs/tags/：标签指向的 commit。

仓库管理

git仓库有三个主要的管理区域。

第一个是工作区，指当前在电脑上面修改文件的区域，这个区域里面文件的修改信息还没有记录在git里面。

第二个是暂存区。暂存区是工作区和版本区过渡的一个区域，便于临时舍弃某个文件的修改，或是控制某些修改，可以通过git status来查看状态。想让文件的修改进入这个区域，需要通过git add指令来进行。git add <file>能够让特定文件的修改进入暂存区，git add . 能够让当前仓库内所有文件的修改进入暂存区。

第三个是版本区。这个区域就是所有通过git commit -m "<update info>"提交的信息所存放的地方。使用git log可以查看这个区域的信息。

撤销操作

实际操作中，我们可能会有很多误操作，这个时候需要我们进行撤销。

  • 在工作区对某文件进行错误修改时，可以使用git checkout -- <filename>来撤销指定文件的修改，还原为上一次commit的状态。
  • 在误提交错误文件到git管理时(例如某个超大文件)，可以使用git rm <filename> && git commit -m "rm file"来删除工作区+版本区的文件，若不想在工作区删除该文件，可以使用git rm --cached <filename>来代替。
  • 在错误commit，想回退版本时，可以使用git reset HEAD~1，这将让HEAD指向上一个提交，但不会删除修改。如果想去掉修改的话，可以使用git reset --hard HEAD~1来代替。
git reset？

git reset在排查错误的时候是非常有用的，能够帮我们定位bug发生在哪个版本里面。此处简单介绍git reset的一些功能。

简单来说，git reset是使工作区的指针HEAD指向版本仓库里面的特定版本，也就是之前某次提交后的项目。

HEAD表示当前版本，HEAD^表示上一个版本，HEAD^^表示上上版本，以此类推。前100个版本就写不过来了，当然这还有更简单的方法HEAD~x，例如HEAD~100能够表示前100版本。

也可以返回特定版本号git reset 1a2b3d。

git reset --hard: 工作区和缓存区的内容全部扔掉！然后返回指定版本。

git reset --soft: 保留工作区内容，并且已提交的内容于当前版本和返回版本的差异会存进缓存区。Attention:存进缓存区的方式是覆盖而不是追加！这意味着你之前存进缓存区的内容会丢失，所以不要连续使用两次git reset --soft。

git reset (--mixed): 保留工作区内容，并且当前版本和返回版本的差异会追加存入工作区。

查看日志，之前的日志git log，之后的日志git reflog。

新的分支

  • 创建新的分支时，可以使用git branch <name>
  • 查看所有分支时，可以使用git branch -a
  • 创建新的分支+切到此分支时，可以使用git checkout -b <name>
  • 切到某个分支时，可以使用git checkout <name>
分支命名？

规范的分支命名在一个项目中非常重要，保证了项目的稳定性和多人开发合作潜力。

1.主分支(main/master): 存放稳定的、可发布的代码。不应该直接在主分支上开发，合并其他分支的代码进行更新。

2.开发分支(develop): 用于集成开发中的功能分支。

3.功能分支(feat):通常命名为feature/<feat-name>或feat/<feat-name>，用于开发新的功能，完成后可合并于develop分支，或是直接抛弃。

4.修复分支(fix):通常命名为bugfix/<...>或fix/<...>，从develop分支创建，修复完成后合并回develop分支。

5.热修复分支(hotfix):通常命名为hotfix/<...>，用于紧急修复生成环境的bug，从main分支创建，修复完成后合并回main和develop分支。

6.发布分支(release):通常命名为release/v1.0.0或release/2025-09-02，从develop分支创建、发布完成后，合并到develop和main分支。

7.支持分支(support):通常命名为support/v1.0.x，从main分支创建，用于维护旧版本。

合并

  • 最简单的合并两个分支，git merge <sub-branch>
  • 合并两个分支，但保留合并基分支的历史的线性化，不零碎掺杂合并从分支的提交历史，git merge --no-ff <sub-branch>。推荐主分支使用。
  • 远程拉取合并分支，git pull origin <branch>相当于git fetch + git merge，此指令如果远程和本地有不同的提交历史，会创建一个新的合并提交，导致提交历史出现分叉。
  • 远程拉取合并分支but rebase，git pull --rebase origin <branch>能够把远程分支的更改拉取到本地，然后将本地的提交一个个重新应用到远程分支的最新提交上，保持提交历史的线性化。推荐使用。

远程管理

  • 添加远程仓库的URL， git remote add <remote-name> <remote-url>，并通过git remote -v确认是否添加。
  • 推送当前分支到远程仓库，并与远程分支关联，git push -u origin <branch>，其中origin 为<remote-name>。
  • 推送当前分支到远程仓库，并与远程分支合并，git push origin <branch>，其中origin 为<remote-name>。
  • 推送当前分支到远程仓库，并将远程分支覆盖，git push -f origin <branch>，其中origin 为<remote-name>。
  • 删除远程分支，git push --delete origin <branch>
  • 抓取远程分支，不合并，git fetch origin <branch>，一次性拉取远程所有分支，git fetch --all

commit规范

事实上，我们的内核项目commit注释一直没有一个统一的规范，进行的比较随意，甚至当成了ctrl+s功能，想保存的时候就提交一下，这导致了我们项目后期debug的一些困难。因此，这里总结一些commit的注释规范，以为后来的项目做准备。

Implement real-time suggestions for search input field.
Fetch top 5 matching results from backend API as user types.
Enhance UX with keyboard navigation support.
 
Closes #101

抽象出来，由以下几部分构成：

  1. 类型（type）: 表示提交的类别。
  2. 范围（scope，可选）: 指定更改影响的模块或范围，用括号包裹。
  3. 主题（subject）: 简短描述更改内容，通常以动词开头，50-72 字符以内。
  4. 正文（body，可选）: 详细说明更改的原因和细节，每行不超过 72 字符。
  5. 脚注（footer，可选）: 关联问题或其他元信息，例如 issue ID。
